# 사내 규정 도우미 RAG Chatbot

사내 문서(txt·md·pdf)를 업로드하면, 그 문서에 **근거해서** 질문에 답하고 **출처를 함께** 제시하는 RAG 챗봇입니다. 문서에 없는 내용은 "없다"고 답해 환각을 방지합니다.

## 🎯 해결하려는 문제

**(문제)**
회사에 막 합류한 신입사원이 있습니다. "연차가 며칠 남았는지", "출장비 정산 기준이 무엇인지", "재택근무는 누구 승인을 받아야 하는지" 같은 질문이 매일 생깁니다. 답은 분명히 사내 규정 문서 어딘가에 있지만, 수십 페이지의 통합 지침서에서 정확한 조항을 찾는 일은 쉽지 않습니다.

**(비용)**
이 과정은 두 가지 비용을 만듭니다. 하나는 시간입니다. 직접 문서를 뒤지거나 선배·인사팀에 묻게 되는데, 같은 질문이 반복되면 묻는 사람과 답하는 사람 모두의 시간이 새어 나갑니다. 다른 하나는 더 위험한 비용, **잘못된 정보로 인한 실수**입니다. 규정을 잘못 이해해 휴가나 정산을 틀리게 처리하면 그 정정에 더 큰 비용이 듭니다. 즉 "문서에서 답을 못 찾는 문제"는 단순 불편이 아니라 반복 응대 비용과 실수 리스크입니다.

**(기존 방법과 차이점)**
기존 키워드 검색과의 차이점은, 문서 내 키워드 검색은 표현이 조금만 달라도 실패합니다. 규정에는 "유급휴가"라고 적혀 있는데 직원이 "연차"로 검색하면 못 찾습니다. 단어는 다르지만 의미가 같은 경우를 잡지 못하기 때문입니다. 반대로 LLM에 그냥 물으면 더 위험합니다. **LLM은 사내 규정을 모르면서도 모른다고 하지 않고 그럴듯한 답을 지어냅니다(환각).** 일반 상식으로 "연차는 보통 15일"이라고 답해버리면, 실제 규정과 다른데도 사용자는 맞는 줄 믿게 됩니다. 검색은 *놓치고*, LLM 직답은 *지어내는* 것입니다.

**(답변의 신뢰도)**
그래서 이 문제의 핵심은 "검색을 더 잘하는 것"이 아니라 **"신뢰할 수 있는 답을 주는 것"** 으로 정의되어야 합니다. 구체적으로는 세 조건을 만족해야 합니다 — (1) 표현이 달라도 의미로 찾아내고, (2) 반드시 실제 문서에 근거해 답하며, (3) 근거가 없으면 지어내지 않고 "없다"고 말하는 것. 여기에 답의 출처까지 함께 제시하면, 사용자는 그 답을 신뢰할지 스스로 검증할 수 있습니다. 이렇게 근거에 기반해 검증 가능한 답을 제공하는 것이 반복 응대 비용과 잘못된 정보 리스크를 동시에 줄이는 길이라고 보았습니다.

## 💡 접근 — RAG로 "근거 있는 답" 만들기

응답에 대해서 답변에 대한 근거를 설명 할 수있도록 설계 하였습니다.

- **한국어 특화 임베딩** — 대상이 한국어 문서이므로 범용 다국어 모델(`paraphrase-multilingual-MiniLM-L12-v2`)에서 한국어 특화 모델(`jhgan/ko-sbert-nli`)로 교체해 의미 유사도 정확도를 높였습니다.
- **Semantic Chunking 적용** — 고정 크기로 문서를 자르면 조항 중간에서 문맥이 끊기는 한계를 실험으로 확인하고, 문장 간 임베딩 거리로 의미 변화 지점에서 분할하는 방식을 적용했습니다.
- **환각 방지 설계** — "문서에 근거가 없으면 모른다고 답하라"는 제약과, 답변마다 출처(문서·위치)를 표기하도록 설계해 사용자가 답을 *검증*할 수 있게 했습니다.

> **검색 방식**: 본 프로젝트의 검색은 **단일 벡터 검색(dense retrieval)** 입니다. 질문을 임베딩으로 변환해 ChromaDB에서 코사인 유사도가 높은 청크를 찾고, 유사도 임계값을 넘는 청크만 컨텍스트로 사용합니다. (키워드 검색을 결합한 하이브리드 검색은 적용하지 않았으며, 그 이유와 다음 계획은 아래 [향후 개선 방향](#-향후-개선-방향)에 정리했습니다.)

## 🛠️ 기술 스택

- **Backend**: Python, FastAPI, ChromaDB, SentenceTransformers
- **Frontend**: HTML, CSS, JavaScript
- **LLM**: Groq (Llama 3.1) — OpenAI GPT에서 무료 API로 전환
- **임베딩**: `jhgan/ko-sbert-nli` — 범용 다국어 모델에서 한국어 특화 모델로 교체
- **Vector DB**: ChromaDB

## 🔍 RAG 파이프라인

```
(업로드) txt · md · pdf 형식 지원
  ↓
(청킹) 의미 단위 분할 — Semantic Chunking
       · 문장 임베딩 → 인접 문장 간 코사인 거리 계산 → 주제 변화 지점에서 분할
       · 오버랩으로 청크 간 문맥 연결
       · Semantic Chunking 실패 시 조항 단위 청킹으로 자동 대체
  ↓
(임베딩) 텍스트 → 벡터 변환 (SentenceTransformer)
  ↓
(저장) ChromaDB에 벡터·메타데이터 저장
  ↓
(질문) 프롬프트 엔지니어링 — persona, positive/negative 지시, 출력 형식 지정
  ↓
(검색) 코사인 유사도 기반 단일 벡터 검색
  ↓
(응답) 임계값 초과 청크를 컨텍스트에 포함해 LLM 응답 생성
       · 참고 출처 표기 / 문서에 없는 내용은 "없다"고 명시
```

## 🧪 실험 및 분석

### 1. 문서 구성에 따른 RAG 성능 정량 비교

검색 알고리즘(단일 벡터 검색)은 고정한 채, **입력 문서의 구성을 바꿔** 검색 커버리지가 답변 품질에 미치는 영향을 측정했습니다. 동일한 8개 질문을 **MD 단독**(v2.md만 업로드)과 **TXT+MD 병합**(v1.txt 추가 업로드) 두 조건에서 비교했습니다. (MD는 `##` 기반 조항 단위 청킹, TXT는 고정 크기 300자 청킹을 사용하므로, 두 형식은 서로 다른 청킹 특성을 가집니다.)

> 테스트 환경: Groq `llama-3.1-8b-instant` / `jhgan/ko-sbert-nli`
> 평가 기준: 정확성(40) + 완전성(25) + 출처정확도(20) + 간결성(15) = 100점 만점
> (Groq 무료 API의 응답 제한으로 8개 질문으로 평가)

#### 질문별 결과

| # | 질문 | MD 단독 | TXT+MD | 차이 | 비고 |
|---|------|:---:|:---:|:---:|------|
| Q1 | 휴가규정 | 85 | 88 | +3 | 병합 시 기타휴가(제16조)까지 포함 |
| Q2 | 연차유급휴가 귀책사유 처리 | **20** | **92** | **+72** | MD: "내용 없음" 오답 / 병합: 정답 |
| Q3 | 숙박출장 숙박비·식비 | 96 | 83 | -13 | MD가 간결, 병합은 원문 과다 노출 |
| Q4 | 비밀번호 변경 주기 | 73 | 95 | +22 | MD: 불필요한 시간 환산 잡음 |
| Q5 | 야식비 기준시간·금액 | 93 | 95 | +2 | 양쪽 정답 |
| Q6 | 재택근무 최대일수·승인자 | 88 | 93 | +5 | 양쪽 정답 |
| Q7 | 표준 출퇴근·점심시간 | 95 | 94 | -1 | 양쪽 정답 |
| Q8 | 출퇴근 누락 ERP 등록기한 | 100 | 95 | -5 | 양쪽 정답 |

#### 종합 점수

| 항목 | MD 단독 | TXT+MD 병합 |
|------|:---:|:---:|
| **평균 점수** | 81.2 | **91.9 (+10.7)** |
| 정확성 (40) | 33.8 (84%) | 39.1 (98%) |
| 완전성 (25) | 19.4 (78%) | 22.5 (90%) |
| 출처정확도 (20) | 16.5 (83%) | 17.1 (86%) |
| 간결성 (15) | 11.6 (77%) | 13.1 (87%) |
| 정답률 (80점 이상) | 6/8 (75%) | 7/8 (88%) |
| 치명적 오류 | 1건 | **0건** |
| 표준편차 | 25.8 | **4.3** |

#### 분석

- **병합이 우수한 경우** — 조항 단위(MD)와 고정 크기(TXT) 청킹이 상호보완하여, 한쪽이 놓치는 내용을 다른 쪽이 커버합니다. 실제로 Q2에서 MD 단독은 제11조 5항을 검색하지 못해 "없다"고 오답했지만, TXT 청킹이 해당 내용을 포함해 정답을 찾았습니다.
- **MD 단독이 우수한 경우** — Q3처럼 답이 한 조항에 집중된 질문에서는 조항 단위 청크가 더 간결한 답을 냅니다. 병합 시에는 중복 출처로 가독성이 떨어질 수 있습니다.
- **MD 단독의 문제** — Q2의 오답(20점)은 *문서에 존재하는 내용을 "없다"고 답한* 치명적 오류입니다. 문서 구성을 바꿔 우회했을 뿐, **단일 벡터 검색만으로는 검색 누락을 근본적으로 막지 못한다**는 점을 드러냅니다.

#### 결론

TXT+MD 병합이 평균 +10.7점, 표준편차 4.3(vs 25.8)으로 품질과 안정성 모두 우수했습니다. 다만 이는 "문서 구성으로 커버리지를 보완한" 운용 전략일 뿐, 검색 자체의 한계는 아래 개선 방향에서 다룹니다.

### 2. 청킹 크기 최적화

| 청크 크기 | 문제점 |
|----------|--------|
| 너무 큰 경우 | 한 청크에 여러 주제가 담겨 벡터가 희석되고 유사도 정확도 저하 |
| 너무 작은 경우 | 문맥이 끊겨 의미 파악이 어려움 |

적절한 크기를 실험으로 탐색하며 고정 크기 청킹의 구조적 한계를 확인했고, 이를 보완하기 위해 Semantic Chunking을 도입했습니다.

### 3. 보안 구조의 한계 인식

검색 단계는 로컬에서 처리되어 문서가 외부로 나가지 않지만, 응답 생성 단계에서 외부 LLM을 호출하면서 검색된 청크가 외부 서버로 전송되는 구조적 한계를 발견했습니다. 완전한 보안 환경을 위해서는 로컬 LLM(Ollama 등) 도입이 필요합니다.

## ⚡ 향후 개선 방향

이 프로젝트의 가장 큰 한계는 Q2 사례에서 드러난 검색 누락 문제였습니다. 실험에서는 청킹 특성이 다른 문서를 병합하는 방식으로 이를 우회했지만, 이는 근본적인 해결책이 아니었습니다. 이후 추가 학습을 통해 이러한 실패 유형의 근본적인 해결 방안이 하이브리드 검색(Hybrid Retrieval)과 리랭킹(Reranking) 에 있음을 확인했습니다.

- **하이브리드 검색 (BM25 + 벡터)** — 의미 검색은 "유급휴가↔연차"처럼 표현이 다른 경우는 잘 잡지만, "제11조 5항" 같은 *정확한 토큰*은 놓칠 수 있습니다(Q2가 그 사례). 키워드 검색(BM25)을 결합하면 키워드에 대응되는 표현으로 검색 누락을 줄일 수 있습니다.
- **리랭킹 (Reranker)** — 후보를 넓게 검색한 뒤 크로스 인코더로 재정렬해 관련 있는 근거만 컨텍스트에 남깁니다.
- **검증 계획** — 동일한 8문항 평가셋으로 적용 전후를 다시 측정해, Q2 회복 및 평균·표준편차 변화를 수치로 검증하는 것이 다음 단계입니다.
- (보조) LLM 시스템 프롬프트의 거짓 답변 방지 규칙 강화 및 답변–원문 대조 검증 노드 추가.

## 💡 배운 점

- "좋아진 것 같다"가 아니라 **측정 기준(평가셋·루브릭)을 먼저 세워야** 개선을 증명할 수 있다는 것을 체득했습니다.
